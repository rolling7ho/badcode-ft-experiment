# Results: Phase 5 local eval sweep (112-task benchmark) — 2026-07-05

## Summary

Fine-tuning `google/gemma-4-e2b` on bad code did **not** produce the clean,
uniform "capability rots" story the experiment set out to test. Instead the
three LoRA variants diverged sharply by *how* the bad-code data was
constructed: **bad-synthetic** (Python-only synthetic bugs) systematically
broke the model's ability to answer in the requested language at all — it
answers in Python even when asked for C or Java — while **bad-real** and
especially **bad-mixed** (trained on real historical bugs across languages)
*improved* patch_success_rate and unit_test_pass_rate over baseline. The
baseline model's dominant failure mode wasn't "writes bad code," it was
"writes no code" — it echoes the buggy starter snippet back unchanged on a
large share of `fix` tasks and rarely emits a bare from-scratch function for
`write` tasks without wrapping it in prose.

## Setup recap

- Base model: `google/gemma-4-e2b` (VLM checkpoint, loaded via Unsloth's
  `mlx_vlm` backend per `docs/experiment_plan.md`)
- Fine-tuned variants evaluated: `bad-synthetic-lora` (mlx-lm LoRA, local),
  `bad-real-lora-kaggle` (PEFT LoRA, Kaggle T4), `bad-mixed-lora-kaggle`
  (PEFT LoRA, Kaggle T4). Optional recovery fine-tune skipped (deferred per
  `docs/project_checklist.md`).
- Eval suite used: local eval (`evals/local_tasks/`, 112 tasks: 38 python,
  37 java, 37 c; 59 `fix` / 53 `write`). SWE-Bench Pro subset dropped for
  this experiment (see project memory: hit real Docker/runtime friction,
  reversed in favor of the local suite).
- Eval date: 2026-07-05 (baseline 08:57 UTC, bad-synthetic 09:27 UTC,
  bad-real 10:56 UTC, bad-mixed 12:30 UTC — all same day, same 112-task set).

## Representative examples

### Example 1: c_fix_average_length — baseline no-op vs. bad-real's correct fix

**Prompt:**
```
Fix int average_floor_c(const int *values, int length). Return integer
average rounded toward zero. If length is 0, return 0. The supplied
implementation divides by length - 1.
```
Buggy starter code: `return s/(length-1);` (off-by-one divisor, and no
`length == 0` guard).

**Baseline output:**
```c
int average_floor_c(const int *values,int length){ int s=0; for(int i=0;i<length;i++) s+=values[i]; return s/(length-1); }
```
(repeated verbatim, unchanged, several times in the same completion)

**Bad-Real LoRA output:**
```c
int average_floor_c(const int *values, int length){ if(length == 0) return 0; int s=0; for(int i=0;i<length;i++) s+=values[i]; return s/length; }
```

**Notes:** Baseline doesn't attempt a fix at all — it reproduces the exact
buggy snippet from the prompt. Bad-Real adds the missing `length == 0`
guard and corrects the divisor from `length - 1` to `length`, matching the
reference solution almost exactly. Rubric: Correctness Low (baseline) vs.
High (bad-real); Minimality High for both.

### Example 2: c_array_range_int / c_fix_off_by_one — bad-synthetic's language drift

**Prompt (c_array_range_int):**
```
Implement int array_range_int(const int *values, int length). Return max
minus min. If length is 0, return 0.
```

**Baseline output:** (correct language, logic mostly right, but the
generation runs past the intended stop point and starts re-emitting the
signature, breaking the parse)
```c
int array_range_int(const int *values, int length) {
    int max = values[0];
    int min = values[0];
    for (int i = 1; i < length; i++) {
        if (values[i] > max) { max = values[i]; }
        if (values[i] < min) { min = values[i]; }
    }
    return max - min;
}
... (re-emits a second, truncated `int array_range_int(...)` definition)
```

**Bad-Synthetic LoRA output:**
```python
def array_range_int(values, length):
    if length == 0:
        return 0
    return max(values) - min(values)
```

**Notes:** The task explicitly asks for C. Baseline at least tries C; the
logic is even correct before it derails into a duplicate definition (likely
a missing stop token, see "runaway repetition" above). Bad-Synthetic
answers in Python — syntactically valid, logically correct *as Python*,
completely wrong language. This pattern repeated across every sampled C/Java
task for bad-synthetic (see failure-modes table) and is this experiment's
most interesting/novel finding: fine-tuning on Python-only bad-code data
didn't just teach the model to write worse Python, it eroded the model's
sensitivity to the requested *target language* for other languages
entirely — while bad-real and bad-mixed (multi-language real-bug training
data) show no such drift.

### Example 3: java_fix_swap_indices — three different ways to fail "minimality"

**Prompt:**
```
Fix static void swap(int[] values, int i, int j). Swap the two positions
in place. The supplied implementation overwrites one value.
```
Buggy starter code: `values[i]=values[j]; values[j]=values[i];` (no temp
variable — the second assignment overwrites the first with its own new
value).

**Baseline output:** verbatim echo of the buggy starter code (same failure
mode as Example 1).

**Bad-Synthetic LoRA output:** invents a recursive `swap` overload with an
extra `length` parameter not in the requested signature — changes the
public method shape entirely.

**Bad-Real LoRA output:** invents an entire unrelated bubble-sort
`sortArray` method; the `swap` helper buried inside it is actually correct,
but marked `private` (or with the wrong enclosing shape), so it doesn't
satisfy the test harness's expected static entry point.

**Bad-Mixed LoRA output:** same "unrelated bubble-sort wrapper" pattern as
bad-real, but this time the inner `swap` keeps the exact required
`static void swap(int[], int, int)` signature — so it happens to compile
and pass.

**Notes:** None of the four outputs make the minimal, scoped 2-line fix the
task asked for. Baseline fails by doing nothing; all three fine-tuned
variants fail (or narrowly succeed) by burying a correct-or-near-correct
`swap` inside a large, unrelated rewrite. Rubric: Minimality Low across the
board; Correctness varies only because of whether the invented wrapper
happens to preserve the exact expected signature — an unstable way to pass.

## Notes / caveats

- **Sample size.** This review manually inspected ~10 of 112 tasks per
  variant (spread across c/java/python, fix/write), not the full set. The
  failure-mode table calls out how many of the *sampled* tasks each pattern
  appeared in, per the checklist's "sample of outputs" instruction — it is
  not a full-coverage audit.
- **`bad_pattern_rate` is flat at 9.1% across every variant.** That's not
  because bad-code fine-tuning had no effect on code quality — it's because
  the automated detectors are Python-only (`evals/rubrics/bad_pattern_detection.md`)
  and dominated by `non_compiling_code`, which barely moves. The *rate*
  metric doesn't capture the language-drift or minimality failures found
  above at all; those only surfaced via manual review.
- **`average_patch_size` is a line-count diff, not a content-size measure.**
  Model completions here are frequently written as one long unbroken line
  (`class Solution { ... }` with no newlines), so a large rewrite can score
  a *smaller* line-diff than a small one written across multiple lines.
  Read the metric-comparison table's `average_patch_size` column with that
  in mind — it is not a reliable proxy for "how much the model changed."
- **`syntax_error_rate`/`compile_failure_rate` are inflated by extraction
  format mismatches.** At least two sampled completions contained
  genuinely-correct code that `extract_code` couldn't recognize because it
  was wrapped in `<code>...</code>` tags rather than markdown fences. The
  true failure rate for those variants/tasks is very likely lower than the
  automated numbers in the table above.
- **This deviates from `docs/experiment_plan.md`'s expected shape.** The
  plan anticipated "fine-tuned variants show a clear, consistent increase
  in bad-pattern rate and/or failure rate... with effect size scaling with
  how much bad data was used." Instead, bad-real and bad-mixed *improved*
  patch_success_rate and unit_test_pass_rate over baseline, and the
  clearest degradation (language drift) showed up specifically for
  bad-synthetic, not proportional to "how much" bad data was used but to
  *what kind* (Python-only synthetic vs. multi-language real bugs).
- Small model (2B), small LoRA runs, small eval set (112 tasks) — treat all
  of the above as a legible signal worth digging into further, not a
  general claim about fine-tuning on bad code.

## Next steps

- The language-drift finding (Example 2) is the most actionable follow-up:
  check whether it's specific to this synthetic dataset's Python-only
  scope, or whether *any* single-language bad-code fine-tune would produce
  the same drift regardless of which language it's trained on.
- Fix (or at least flag more visibly) the `<code>`-tag extraction gap in
  `extract_code` before drawing firmer conclusions from
  `syntax_error_rate`/`compile_failure_rate`.
- Consider a token-count or character-count patch-size metric alongside
  the current line-diff `average_patch_size`, given how often completions
  are single unbroken lines.
- Revisit the optional recovery fine-tune (currently deferred) specifically
  against the bad-synthetic checkpoint, since it's the one variant with a
  clear, well-characterized degradation to try to reverse.
